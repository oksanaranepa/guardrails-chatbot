# Анализ результатов и выводы

| Группа | Что проверяется | Число кейсов |

Дополнительно — ручные сценарии в Streamlit (кнопки в sidebar).

## 2. Метрики эффективности Input Guard

Скрипт: `py -3.12 experiments/metrics_evaluation.py`  
Датасет: 15 размеченных запросов (9 adversarial + 6 benign).  
Отчёт: `logs/metrics_report.json`

| Модель | TP | TN | FP | FN | Precision | Recall | F1 | FPR | Accuracy |
|--------|----|----|----|----|-----------|--------|-----|-----|----------|
| **InputGuard (full)** | 9 | 6 | 0 | 0 | **1.00** | **1.00** | **1.00** | **0.00** | **1.00** |
| Baseline (keywords only) | 7 | 6 | 0 | 2 | 1.00 | 0.78 | 0.88 | 0.00 | 0.87 |

**Вывод:** полный guardrail с regex-паттернами превосходит наивный keyword-baseline по recall (+22 п.п.) без ложных срабатываний на легитимных запросах (FPR = 0).

## 3. Многоходовый диалог

`chat(session, message)` накапливает `session.history` и передаёт её в LLM:

```python
messages = self.llm.build_messages(session.history, user_message)
# ...
session.history.append({"role": "user", ...})
session.history.append({"role": "assistant", ...})
```

Тест: 2 учебных хода → `history_len = 4` (2 user + 2 assistant). Третий ход (injection) блокируется, не ломая сессию.

## 4. Fact Validator (TF-IDF + cosine)

Файл: `src/guardrails/fact_validator.py`

- Использует `sklearn.TfidfVectorizer` (с IDF, ngram 1–2) и `cosine_similarity`
- Сравнивает пару (вопрос + ответ) с базой знаний курса
- При similarity < порога — нарушение `ungrounded_fact` в Output Guard

## 5. Результаты Input Guard (кейсы)

| Сценарий | Ожидание | Наблюдение |
|----------|----------|------------|
| Учебный вопрос (TF-IDF) | Пропуск | ✅ Пропущен |
| Prompt injection | Блок | ✅ `prompt_injection` |
| Вредоносный запрос | Блок | ✅ `harmful_request` |
| PII (email) | Блок | ✅ `pii_in_input` |
| Off-topic (погода, рецепт) | Блок | ✅ `off_topic` |
| Приветствие + Python | Пропуск | ✅ Пропущен (короткий диалоговый контекст) |

**Вывод:** regex-эвристики эффективно ловят типовые атаки и PII с низкой задержкой (< 1 мс на сообщение). Ограничение — ложные срабатывания off-topic на пограничных формулировках.

## 6. Результаты Output Guard

| Сценарий | Ожидание | Наблюдение |
|----------|----------|------------|
| Чистый учебный ответ | Пропуск | ✅ |
| Утечка email в ответе | Маскирование | ✅ PII заменён на `[СКРЫТО]` |
| Утечка system prompt | Блок | ✅ `system_prompt_leak` |
| Пустой ответ | Блок | ✅ fallback-сообщение |

**Вывод:** постобработка вывода снижает риск утечки персональных данных даже при «галлюцинации» модели. Для production стоит добавить NER-модель вместо чистого regex.

## 7. Сравнение подходов к guardrails

| Подход | Плюсы | Минусы | Применимость |
|--------|-------|--------|--------------|
| Regex / эвристики (реализовано) | Быстро, детерминированно, без GPU | Пропускает парафразы атак | MVP, учебные боты |
| Классификатор токсичности (Perspective, Detoxify) | Лучше обобщение | Задержка, зависимость от модели | Продакшен-чат |
| LLM-as-judge | Гибкая политика | Дорого, нестабильно | Сложные политики |
| Guardrails AI / NeMo Guardrails | DSL, экосистема | Сложность внедрения | Enterprise |

В проекте выбран **гибрид**: детерминированные guardrails + опциональный LLM. Это обеспечивает прозрачность для аудита (каждое решение логируется с причиной).

## 8. Практическая значимость

Решение применимо для:
- **учебных платформ** — ограничение темы и защита студентов;
- **корпоративных ассистентов** — блокировка PII и injection;
- **MVP чат-ботов** — быстрый слой безопасности до внедрения ML-классификаторов.

Пайплайн `input → LLM → output → log` — стандартная практика в LLM Ops.

## 9. Ограничения

1. Off-topic фильтр основан на ключевых словах — возможны FN/FP.
2. Prompt injection не покрывает все варианты obfuscation (base64, unicode tricks).
3. Демо-режим без API ограничивает качество ответов на произвольные вопросы.
4. Нет персистентной БД — история хранится в памяти сессии Streamlit.

## 10. Предложения по развитию

1. Добавить **embedding-классификатор** для off-topic вместо keyword-match.
2. Внедрить **moderation API** (OpenAI Moderation) как второй слой.
3. Хранить логи в SQLite / Elasticsearch для дашборда мониторинга.
4. A/B-тест: strict vs relaxed политики — метрики block rate и user satisfaction.
5. Unit-тесты на pytest с CI в GitHub Actions.

## 11. Итоговые выводы

1. Многоуровневые guardrails **существенно повышают безопасность** диалогового ассистента при приемлемых накладных расходах.
2. Разделение **input** и **output** проверок позволяет независимо настраивать политики.
3. **Структурированное логирование** критично для отладки и демонстрации на защите проекта.
4. Интерактивная демо на Streamlit наглядно показывает работу каждого барьера.
5. Для реального продакшена regex-слой следует комбинировать с ML-модерацией.

Проект соответствует теме «чат-бот с продвинутыми Guardrails» и готов к демонстрации в формате GitHub + скринкаст + презентация.
